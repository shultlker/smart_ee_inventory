## 1. 背景与目标

### 1.1 文档范围

本文档描述**最终目标**（64 格 + 射频复用 + 看门狗出入库），并单独说明**当前仅有一块单天线 RFID 开发板**时，如何用现有系统做**可演示的原型**，且不与远期架构冲突。

### 1.2 现状（已实现）

| 能力 | 说明 |
|------|------|
| 硬件 | **1×** YZ-M40 USB 模块 + **1×** 全向/近场天线（COM11 等） |
| 读卡 | 连续盘存 0x21，`event_bus` 推送 |
| 绑定 | EPC 写入 `bin_slots.rfid_tag_epc`；种子 `BIN-TEST` 三格各 1 标签 |
| 入库 | `/inventory/register` 手动选格 + 听卡/手输 EPC；仪表盘未登记引导 |
| **看门狗** | **仅格位物料** → 待确认 `take_out`/`return` → 弹窗 |
| **非标借还** | 仪表盘读卡 → `asset_confirm` 弹窗（不看门狗） |
| **标签管理** | `/inventory` 标签区：绑定/换绑/解绑/删除 |
| **双类型** | `slot_material` + `asset`（`assets` 表） |
| **仍缺** | 64 路物理格位定位、射频复用器驱动 |

### 1.3 远期目标（64 路硬件到位后）

| 目标 | 说明 |
|------|------|
| **格位级定位** | 复用器第 *k* 路接通时读到的 EPC → 第 *k* 格 |
| **两类库存** | 格位物料（看门狗）+ 非标资产（手动读卡借还） |
| **看门狗** | EPC 出现/消失 → 对话框 + 借还/消耗流水 |
| **可追溯** | 使用人、项目、未知 EPC 引导绑定 |

**实现顺序**：先做 **§3 单天线演示** → 再按 §10 路线接 64 路硬件。

## 2. 当前硬件：单天线开发板

### 2.1 实物拓扑（现在）

```
  PC (main.py)
      │ USB 串口
      ▼
  YZ-M40 读写器 ── RF 馈线 ── 【唯一天线】
                              │
                         桌面读卡区 / 手工靠近
                         （无法区分 64 格物理位置）
```

**结论**：在复用器到位之前，**不能**靠 RF 自动得知标签在第几格；格位只能来自：

1. **数据库既有绑定**（种子 EPC ↔ `bin_slots`）；
2. **用户在 UI 上先选格再读卡**（入库绑定页已有）；
3. **软件仿真**（虚拟通道号，仅供演示 64 格 UI）。

### 2.2 单天线的能力边界

| 能做 | 不能做（需 64 路近场） |
|------|------------------------|
| 识别 EPC、RSSI | 自动判断标签在哪个格子 |
| 检测标签**在读卡区内/外**（看门狗） | 64 格**同时**在场内监控 |
| 已绑定 EPC → 查格位/物料 | 相邻格**零串读** |
| 未知 EPC → 引导入库 | 无人工介入的格位轮询 |

### 2.3 演示用读卡区建议

- 天线固定于**桌面垫**或**小支架**，读距 3～8 cm（可调模块功率/天线类型）。
- 贴标签的物料**单独**拿进/拿离读卡区，避免多标签堆叠（单天线多标签会一次读多个 EPC）。
- 使用种子库 **3 张标签**轮换演示即可（见 §3.2）。

## 3. 单天线演示方案（构思）

> **原则**：用「**一个读卡区** = 一个逻辑 `read_zone`」模拟未来的「某一格天线场」；看门狗只维护**读卡区内 EPC 集合**，不假装有 64 路 RF。

### 3.1 演示模式总览

| 模式 | 代号 | 观众看到什么 | 软件要点 | 硬件 |
|------|------|--------------|----------|------|
| **A. 桌面看门狗** | `portal` | 标签移入/移出垫子上弹出借还/消耗 | 单区 presence 快照 + 对话框 | 仅单天线 |
| **B. 指定格入库** | `manual_slot` | 点格位 → 再读卡 → 绑定入库 | 已有 `/inventory/register` | 仅单天线 |
| **C. 虚拟 64 路轮询** | `sim_mux` | 仪表盘 8×8 格依次「高亮扫描」 | 软件 channel 1..64 计时，不控 RF | 无需复用器 |
| **D. 混合演示** | `demo` | A+C：扫描动画 + 真实读卡区联动 | `RFID_DEMO_MODE=portal` 等配置 | 单天线 |

**推荐对外演示路径**：**B（入库绑定）+ A（格位物料看门狗弹窗）+ 非标手动借还页**；C 仅作 UI 概念展示。

### 3.2 三段式现场脚本（约 5 分钟）

种子数据 `BIN-TEST`（1×3 格）与标签：

| 标签 EPC | 绑定 | 条目 | 演示动作 |
|----------|------|------|----------|
| `E28011704000021CCCF9A58E` | T01-1-1 | 测试电阻 10kΩ | ① 移出 → **出库弹窗**（使用人/项目）② 放回 → **归还弹窗**（消耗） |
| `E28068940000502244813C7D` | T01-1-2 | 陶瓷电容 100nF | `/inventory/operations` 查看记录 |
| `E28011704000021CCCF9A59E` | AST-0001 | Jetson Nano（非标） | 仪表盘读卡弹窗借出/归还 |
| — | T01-1-3 | 红色 LED | 无标签，仅格位库存（BOM 演示） |

**步骤建议**：

1. 打开仪表盘 `/`，选 `BIN-TEST`，展示 3 格货柜网格与库存。
2. 将 **电阻** 标签放入读卡区 → 若已出库则弹出**归还确认** → 填消耗 → 确认。
3. 拿走标签 → **出库确认弹窗** → 填使用人/项目 → 库存 -1。
4. 再次放入 → **归还弹窗** → 消耗可填 0 → 库存恢复或扣减。
5. 切换 **Jetson** 标签 → 仪表盘弹窗借出/归还。
6. `/inventory` 标签区演示绑定/换绑/解绑；`/inventory/operations` 查看记录。

### 3.3 模式 A：桌面看门狗（单天线核心）— **已实现待确认版**

**代码**：`presence_watchdog.py`、`operation_service.py`、`presence_confirm.py`、`lifespan.py`

**逻辑读卡区**内维护在场 EPC 集合，经去抖后创建 **pending** 操作并弹窗：

```
每 T 毫秒（如 200ms）：
  S_curr = 当前盘存到的 EPC 集合
  APPEAR   → pending return（若格位已 checked_out）
  DISAPPEAR → pending take_out（若格位 occupied）
  用户确认 → confirm API → 改库存 + inventory_operations.confirmed
  用户取消 → cancel API → 恢复格位状态
```

**已实现配置**（`config/settings.py` / `.env.example`）：

```ini
RFID_PRESENCE_ENABLED=true
RFID_PRESENCE_APPEAR_COUNT=2
RFID_PRESENCE_DISAPPEAR_COUNT=6
RFID_PRESENCE_TICK_MS=200
RFID_PRESENCE_MISS_GRACE_MS=1200
RFID_PRESENCE_BOOTSTRAP_MS=5000   # 网关就绪后计时；bootstrap 内 tick 不判离开
```

**已实现**：格位物料出库/归还弹窗；非标 **手动借还**；标签 bind/rebind/unbind/delete；操作记录页。

### 3.4 模式 B：指定格 + 读卡（已实现，演示直接用）

`/inventory/register`：

1. 选 `BIN-TEST`、行列或空位；
2. **元件类型与规格**：物料下拉为料号（如 `TEST-LED-RED`），可「新建元件」；
3. 「开始监听 RFID」→ 标签靠近单天线；
4. 确认入库。

**演示话术**：「没有 64 路天线时，格位由人工选定，RFID 只负责身份。」

### 3.5 模式 C：虚拟 64 路（纯 UI / 仿真）

```
for ch in 1..64:
    highlight_grid_cell(ch)      # 仅 UI
    sleep(dwell_ms)              # 如 80ms，整圈 ~5s
    # 若 presence 读卡区有 EPC，可选：写入 log「仿真：若在 ch 路则读到 xxx」
```

- **不连接**复用器；不对 YZ-M40 发切换命令。
- 用于向观众解释**未来**硬件如何轮询；可与真实读卡并行（真实 EPC 仍只来自桌面区）。

### 3.6 非标资产（单天线下的简化演示）

无第二读头时，非标资产与格位物料**共用同一读卡区**：

- 在 DB 增加 `assets` 表前：可用「特殊 part 或 remark=asset」临时演示；
- 演示脚本：贴 Jetson 标签 → 登记为 asset → 拿走触发借出 → 放回触发归还。

### 3.7 单天线演示 vs 远期 64 路对照

| 维度 | 单天线演示（现在） | 64 路近场（将来） |
|------|-------------------|-------------------|
| 位置来源 | DB 绑定 + 人工选格 | 复用器通道号 + DB |
| 看门狗范围 | 1 个读卡区 | 64 个格各自独立（理论上） |
| 同时监控 | 读卡区内所有 EPC | 每格 1～N 个 EPC |
| 演示可信度 | 流程/UI/流水 | 物理防串格 |
| 开发优先级 | **P-Demo：presence + 对话框** | P-Mux：驱动 + 扫描器 |

## 4. 目标硬件架构（远期 · 64 路）

### 4.1 物理结构（料盒柜 8×8 = 64 格）

```
                    ┌─────────────────────────────────────┐
                    │  USB 主机 (PC)                       │
                    └──────────────┬──────────────────────┘
                                   │ USB 串口
                    ┌──────────────▼──────────────────────┐
                    │  YZ-M40 UHF 读写器模块               │
                    └──────────────┬──────────────────────┘
                                   │ 单路 RF 口
                    ┌──────────────▼──────────────────────┐
                    │  64 路射频开关 / 多路复用器板         │
                    │  (SP64T 或级联 SP8T × 2 等)          │
                    └──┬──┬──┬── ... ──┬──┬──┬────────────┘
                       │  │  │         │  │  │
                     Ant1 Ant2 Ant3   Ant62 ... Ant64
                       │  │  │         │  │  │
                    ┌──▼──▼──▼─────────▼──▼──▼────────────┐
                    │  64 格料盒柜（每格底部近场天线/定制 PCB） │
                    │  [1,1] [1,2] ... [8,8]               │
                    └──────────────────────────────────────┘
```

**设计要点**：

- 每格 **1 个近场天线**，辐射范围限制在 **几厘米**（仅覆盖单格），降低串读相邻格。
- 定制 PCB 可将 64 路馈线汇流至复用器，减少手工布线。
- 复用器控制接口需明确：**GPIO / I2C / SPI / 独立 MCU 串口**（实现前必须与实物对齐）。

### 4.2 与 YZ-M40 的关系（远期）

| 项目 | 当前 | 改造后 |
|------|------|--------|
| 天线数 | 1 | 64（同一时刻仅 1 路接通） |
| 盘存方式 | 连续 0x21 盘存 | **按天线切换**：切天线 → 稳定延时 → 短时盘存 → 停盘存 → 下一格 |
| 串口 | 仅 YZ-M40 | 可能 **+1 路** 控制复用器（或 GPIO 扩展板） |
| EPC→位置 | DB 查表绑定 | **轮询通道号** + 可选 DB 绑定（双因子） |

### 4.3 非标物件读取（第二类库存 · 远期）

非标资产（电动工具、Jetson、摄像头）**不一定**放在 64 格柜内，常见方案：

| 方案 | 描述 | 适用 |
|------|------|------|
| **A. 专用「资产区」天线** | 复用器预留 1～N 路接桌面/柜门近场垫 | 固定存放区 |
| **B. 独立门户读写器** | 第二台 YZ-M40 或同一读写器 + 手动切换 | 出入口 |
| **C. 与 64 格共用复用器** | 格 1～60 料盒，格 61～64 资产垫 | 硬件最省 |

软件上应抽象为 **`read_zone`（读取区域）**，不与「8×8 料盒格」强绑定。

### 4.4 硬件风险与验证项（64 路阶段）

| 风险 | 缓解 |
|------|------|
| 近场仍串读相邻格 | 调低功率、加隔板、测 RSSI 阈值；软件可要求「仅当前通道 + RSSI > T」 |
| 复用器切换瞬态误读 | 切换后 **稳定等待**（如 5～20 ms）再开 0x21；切换前先 0x23 停盘存 |
| 64 格全扫周期过长 | 估算：`(切换延时 + 盘存窗口) × 64`；可只扫「活跃格」或分层扫描 |
| 同一格多标签 | 近场格内可能有多卷带/多盘；需支持 **1 格 → 多 EPC** |
| 复用器与读写器阻抗失配 | 硬件 RF 工程师标定 VSWR / 插入损耗 |

## 5. 业务概念：两类库存

### 3.1 类型定义

| 类型 | 代码建议 | 物理形态 | 标签贴在哪 | 典型操作 |
|------|----------|----------|------------|----------|
| **格位物料** | `slot_material` | 卷带、料盘、小袋 | 物料或内包装 | 取出/归还消耗 **数量** |
| **料盒容器** | `bin_container` | 可整盒取走的料盒 | 料盒外壳 | 整盒 **借出/归还** + 项目登记 |
| **非标资产** | `asset` | 工具、开发板、相机 | 设备本体 | **借出/归还/报修** |

当前数据库中：

- `parts` + `inventory_items` + `bin_slots` 已覆盖 **格位物料** 的大部分；
- `bin_cabinets.rfid_tag_epc` 可表示 **料盒容器** 级标签；
- **非标资产** 需扩展（见 §5）。

### 3.2 「料盒取出」vs「格内物料消耗」

用户举例中的两种语义需区分：

```
料盒 xx 取出     → 容器级 EPC 从某「槽位/货架」消失 → 借出登记（人、项目）
料盒 xx 归还     → 容器 EPC 出现 → 可提示「消耗内容物 N 件」（格位库存扣减）
格位 3 标签消失  → 该格绑定的物料 EPC 不见 → 可能取走一卷电容
```

软件看门狗应对 **EPC** 做跟踪，再根据 EPC 绑定的 **实体类型** 分支 UI。

## 6. 看门狗（Presence Watchdog）逻辑

### 6.1 核心思路

> **单天线演示**：仅维护 **1 个读卡区** 的 `Set<EPC>`，无 `antenna_index`；格位由 EPC 绑定表反查。  
> **64 路远期**：维护 `channel 1..64` 各一组 EPC，再合并或分 channel 做看门狗。

维护两张「快照」：

- **按读取区域**：`zone_id × antenna_index → Set<EPC>`
- **全局 EPC 索引**：`EPC → {zone, antenna, last_seen, entity_type}`

每个完整扫描周期（64 路轮询完毕 = 1 frame）更新快照，与上一 frame 比较：

| 事件 | 条件（简化） | 业务含义 |
|------|--------------|----------|
| **APPEAR** | `EPC ∉ prev` 且 `EPC ∈ curr` 连续 **M** 次 | 标签进入场区 |
| **DISAPPEAR** | `EPC ∈ prev` 且 `EPC ∉ curr` 连续 **N** 次 | 标签离开场区 |

**M、N** 为去抖计数（建议默认 M=1～2，N=2～5），避免偶发漏读导致误弹窗。

### 4.2 状态机（单个 EPC）

```
                    ┌──────────┐
         首次见到   │ UNKNOWN  │
        ──────────► │ (未绑定)  │
                    └────┬─────┘
                         │ 绑定入库
                    ┌────▼─────┐
              ┌────►│  ABSENT  │◄────┐
              │     └────┬─────┘     │
     连续 N 次未见到      │           │ 连续 M 次见到
              │     ┌────▼─────┐     │
              └─────│ PRESENT  │─────┘
                    └────┬─────┘
                         │ 触发 APPEAR / DISAPPEAR 事件
                    ┌────▼─────────────┐
                    │ 业务事件队列     │
                    │ → UI 对话框     │
                    └─────────────────┘
```

### 4.3 与现有 `rfid_events` 的关系

| 现有 | 看门狗扩展 |
|------|------------|
| 每条读卡写 `rfid_events`（流水日志） | 保留，但量会 ×64，需 **聚合/采样** 或 WAL |
| 无 disappear | 新增 `rfid_presence_events` 或扩展 event `type`：`appear` / `disappear` |
| 无 antenna_index | 事件需带 `mux_channel`、`cabinet_id`、`slot_id`（推导） |

## 7. 数据模型扩展（建议）

> 以下为设计建议，**未建表**；实现时可 Alembic 或 `create_all` 增量。

### 5.1 读取区域与天线映射

```
read_zones
  id, code, name, zone_type  -- cabinet_64 | asset_bay | portal
  mux_device_id              -- 关联哪块复用器
  cabinet_id (nullable)      -- 若即某 bin_cabinets

mux_devices
  id, code, control_port     -- COM12 / i2c:0x20
  protocol                   -- 厂商协议枚举
  channel_count              -- 64

antenna_channels
  id, mux_device_id, channel_no (1..64)
  cabinet_id, row_no, col_no  -- 映射到逻辑格位
  label, enabled
```

现有 `bin_cabinets.row_count/col_count` 可设为 8×8；`bin_slots` 在创建料盒时 **自动生成 64 条**，并与 `antenna_channels` 一一对应。

### 5.2 非标资产

```
assets
  id, asset_code, name, category  -- tool | dev_board | camera | other
  rfid_tag_epc (unique)
  default_zone_id / default_antenna (nullable)
  status  -- in_stock | checked_out | maintenance
  serial_no, remark

asset_checkouts
  id, asset_id, user_name, project_name
  checkout_at, due_at, return_at
  purpose, remark
```

### 5.3 借还与项目登记

```
checkout_records   -- 统一借还流水（料盒容器 + 资产 + 可选格位物料）
  id, entity_type   -- bin_container | asset | slot_material
  entity_id
  action            -- take_out | return | consume
  user_name
  project_code / project_name
  quantity_delta    -- 归还消耗时
  rfid_tag_epc
  created_at
```

可与现有 `inventory_transactions` 联动：归还消耗时写 `type=out` 流水。

### 5.4 EPC 绑定表（可选，统一入口）

当前 EPC 分散在 `bin_cabinets`、`bin_slots`、未来 `assets`。建议长期统一：

```
rfid_bindings
  epc (PK)
  entity_type, entity_id
  bound_at
```

看门狗只查此表决定弹哪种对话框。

### 6.2 扫描周期伪代码（设计级）

```
for channel in 1..64:
    mux.select(channel)
    sleep(stabilize_ms)
    reader.start_inventory()
    tags = reader.read_for(window_ms)
    reader.stop_inventory()
    snapshot[channel] = tags

watchdog.compare(prev_snapshot, snapshot)
emit events to EventBus + DB
prev_snapshot = snapshot
sleep(idle_ms)
repeat
```

**配置项**（`.env` 扩展）：

| 变量 | 含义 |
|------|------|
| `MUX_ENABLED` | 是否启用多路复用 |
| `MUX_SERIAL_PORT` | 复用器控制口 |
| `MUX_CHANNEL_COUNT` | 64 |
| `MUX_STABILIZE_MS` | 切换后稳定时间 |
| `MUX_DWELL_MS` | 每路盘存窗口 |
| `PRESENCE_APPEAR_THRESHOLD` | M |
| `PRESENCE_DISAPPEAR_THRESHOLD` | N |

### 7.1 对话框触发矩阵

| 事件 | EPC 绑定类型 | 对话框标题 | 必填字段 | 后端动作 |

| DISAPPEAR | 格位物料 | 格位 **{slot}** 物料已取走 | 使用人、项目、数量(可选) | 预扣库存或待归还 |
| DISAPPEAR | 非标资产 | 资产 **{name}** 已取出 | 使用人、项目 | `asset_checkout` |
| APPEAR | 料盒容器 | 料盒 **{code}** 已归还 | 消耗内容物数量（按格位汇总或手动） | `inventory_transactions` 出库 |
| APPEAR | 格位物料 | 格位 **{slot}** 物料放回 | 确认数量 | 调整库存 |
| APPEAR | 非标资产 | 资产 **{name}** 已归还 | 操作：归还 / 报修 | 关闭 checkout |
| APPEAR | **未绑定** | 检测到新标签 | 操作：入库绑定 / 登记资产 / 忽略 | 跳转 register 或 asset 创建 |

### 7.2 全局看门狗 UI 原则

1. **非阻塞队列**：短时间多个 EPC 变化 → 事件队列，逐条处理，避免对话框叠 10 层。
2. **可 snooze**：「稍后处理」入队，仪表盘显示待办角标。
3. **现场复核**：对话框展示 EPC 短码、RSSI、天线号/格位号，便于人工纠错。
4. **权限（远期）**：使用人默认当前登录用户；v0.1 可文本输入。

## 11. 性能与规模估算

假设：`STABILIZE=10ms`，`DWELL=50ms`，64 路：

- 单帧扫描 ≈ `64 × (10+50) = 3.84s`（不含协议开销）
- 看门狗 N=3 → 消失确认 ≈ **11.5s 量级**
- 若需 **<2s** 响应，需：减少 DWELL、并行读（多读写器）、或只扫变化通道

**数据库**：每帧每格多条 EPC × 持续写入会放大 SQLite 锁竞争 → 实现阶段应 **WAL + 批量写 + 快照表与原始读卡分离**。
